Explain Data Concepts

Data types, structures, transformations, schemas, and core database vocabulary

SQL Data Types — Most Missed Topic (7 misses across exams)

CHAR(n)Fixed-length single characters (e.g., account codes "S", "C")Always pads to n — wastes space if lengths vary
VARCHAR(n)Variable-length ASCII/English text (addresses, names)Only uses storage proportional to actual string
NVARCHAR(n)Multi-language text: Chinese, Arabic, Russian, Japanese, GreekN = Unicode/National. Use whenever non-English characters exist

Exam Pattern — The Trick Questions:
"Multi-language / Unicode" → Always NVARCHAR, never VARCHAR.
"Financial / no rounding errors" → Always DECIMAL, never FLOAT.
"Images, videos, binary files" → BLOB. "Large text documents" → CLOB.
"Non-sequential unique ID across servers" → UUID, never INT.

Data Structures & File Formats

.csv
Flat tabular — rows and columns, no nesting. No charts, no formatting. Plain text. Best for raw data exchange.
.json
Hierarchical, nested, human-readable. Lightweight. Standard for web APIs and config files. Supports arrays within objects.
.xml
Hierarchical with nested tags. More verbose than JSON. Good for complex structured data needing strict validation (schemas).
.xlsx
Excel format. Supports charts, pivot tables, conditional formatting. Not lightweight — for analytical reporting, not data interchange.
.dat
Generic, unstructured, application-specific binary or text. No fixed format. Used for game saves, raw program data.
.log
Sequential event records for debugging and monitoring. Text-based, time-ordered. Not for structured data storage.
Key Decision Rule: "Charts + pivot tables + conditional formatting" → .xlsx. "Human-readable + hierarchical + API interchange" → .json. "Raw, no fixed format, application-specific" → .dat. "Tabular only, no formatting" → .csv.

Data Structures (Programming / CS)

StructurePurposeLookup Speed

Linked ListSequential access; each node points to next. Must traverse from the start to find a record.O(n) — Slow

Database Table Types (Data Warehouse)

Fact Table
Stores quantitative measurements and foreign keys. Sales amount, product ID, customer ID, date, store ID. The "what happened" table. Used for aggregation (totals, averages per month/store).
Dimension Table
Stores descriptive context: customer name, product name, store location. The "who/what/where" table. No measurements — just attributes.
Bridge Table
Resolves many-to-many relationships. Holds foreign keys from both sides (e.g., StudentID + CourseID). Also called junction or associative table.
Lookup Table
Maps codes to descriptions (status codes, type codes). Static reference. NOT for many-to-many resolution and NOT for tracking historical changes.
Slowly Changing Dimensions (SCD) Data Warehouse
A data warehousing technique for tracking and preserving historical changes to dimensional attributes (customer address changes, membership level changes from Basic to Premium) while always providing access to the current state. SCDs are used when data evolves slowly over time and you need a history of those changes.

Core Data Concepts

Metadata
"Data about data." Data type, field length, description, source system for attributes like Product_ID. Describes the structure, not the actual business content.
Master Data
Core business entities shared across the org: customer names, product catalogs, supplier lists. The actual business content — not descriptions of fields.
Data Lineage
The complete journey of data from origin through every transformation to its final destination. Used to trace how a metric was derived and to demonstrate compliance during audits.
Data Dictionary
Defines fields, data types, and descriptions for a dataset. Tells you WHAT data is — not where it came from or how it was transformed.

Identify Types of Data Sources

Databases, APIs, storage systems, repositories, and data access methods

Database Types — Know When to Use Each

Database TypeBest ForKey Signal Words
Relational (RDBMS)Structured data, complex joins, ACID transactions, rollback support, referential integrity"enforce consistency," "complex joins," "transaction rollback," "primary/foreign keys"
GraphRelationship-heavy queries: recommendation engines, social networks, fraud detection"relationships between entities," "customers ↔ products ↔ purchases"
ColumnarLarge-scale analytics on sparse data; reads only needed columns"high-speed analytics," "sparse datasets," "semi-structured," "analytical workloads"
Time-SeriesTime-stamped metrics: sensor data, stock prices, performance monitoring"time-stamped," "metrics over time," "IoT," "telemetry"

Critical Distinction: Graph database = relationship traversal (recommendation engines). Relational database = structured queries + ACID + rollback. Columnar = analytics on large datasets. Time-series = time-stamped metrics. These four are frequently tested against each other.

Data Repositories

Data Warehouse
Consolidates data from the entire organization. Optimized for analytics and reporting. Entire org view — not department-specific.
Data Mart
Subject- or department-specific subset of a warehouse. Gives one team (e.g., loan officers) fast, focused access to only their data.
Data Lake
Stores vast raw, unstructured data from all sources. Not curated or optimized for specific queries. Schema-on-read.
Operational DB (OLTP)
Handles day-to-day transactions. Tuned for INSERT/UPDATE speed, not large analytical queries.

API Types

API TypeKey CharacteristicSignal Words
RESTStandard HTTP; returns JSON; fixed endpoint responses; simple and widely used"external service + JSON response," "weather API," "standard request-response"
GraphQLClient specifies exactly which fields to return; single request across multiple sources; reduces over-fetching"specify exact fields," "reduce bandwidth," "single request from multiple sources"
SOAPXML-based protocol; rigid schema; heavy; enterprise legacy systems"XML," "strict schema," "enterprise," NOT JSON
WebSocketPersistent two-way connection; real-time streaming (live chat, live feeds)"persistent connection," "bidirectional," "streaming"
RPCInvokes specific functions on a remote server; fixed endpoints"remote procedure call," "invoke function"
REST vs GraphQL: REST returns a fixed, pre-defined JSON payload. GraphQL lets the client declare exactly which fields it needs — avoiding over-fetching. If the scenario mentions "reduce bandwidth" or "specify fields," it's GraphQL.

Web Scraping Methods

Headless Browser Scraping
Runs a full browser engine (Puppeteer, Playwright) without UI. Executes JavaScript, waits for dynamic content to load. Use when content is rendered by JavaScript after page load.
Static HTML Parsing
Reads raw HTML source (BeautifulSoup). Only sees the initial page load — cannot capture JavaScript-rendered content. Fast and simple for static pages.
Regular Expressions
Pattern matching for extracting text from already-retrieved content. Cannot load pages or execute JavaScript on its own.

DOM (Document Object Model)

The DOM is the browser's in-memory, tree-like representation of an HTML document. Web scraping libraries navigate and query the DOM to locate specific HTML elements (divs, spans, tables) and extract their text content. When the question asks about extracting product prices and descriptions from an HTML page, the answer is DOM — not JSON, not XML.

Identify Infrastructure Concepts

Storage types, cloud models, deployment options, containerization

Storage Types — 4 Misses on Practice Exams

Storage TypeUse CaseKey Signal Words
Block StorageDedicated VM volumes; databases needing low-latency OS-level access. Data split into fixed-size blocks."dedicated volume," "VM storage," "fixed-size blocks," "expand/move between servers," "low latency"
File Storage (NAS)Multiple users sharing files over local network. Office-style shared drives."multiple users on local network," "shared files," "workstations access same files"
Object StorageLarge unstructured files (media, backups) accessed via API. Massive scalability. Metadata tagging."metadata tagging," "no traditional file system," "multi-location access," "S3," "massive scale"
S3 Glacier (AWS)Long-term archival; infrequently accessed backup data. Very low cost, high durability, slow retrieval."archive," "backup," "rarely accessed," "low cost," "high durability"
The Block vs NAS Trick: Block storage = raw OS-level volume for one VM or database. NAS = file-level shared access for multiple office users over LAN. An SSD is a physical hardware component, not a storage architecture — never pick SSD as the answer for a cloud or network storage question.

Cloud Service Models

IaaS
Infrastructure as a Service. Raw compute, storage, networking. Customer still manages OS, runtime, scaling. Most control, most responsibility.
PaaS
Platform as a Service. Provider manages servers, networking, scaling, patching. Developer just deploys app. "Deploy app, don't manage infrastructure."
SaaS
Software as a Service. Fully finished applications (email, CRM). End users, not developers. No deployment, no app building.

Infrastructure Deployment Models

Public Cloud
Minimal upfront cost. Provider manages everything. Scales instantly. Best for startups.
Private Cloud
Dedicated to one org. Higher cost. More control and security.
Hybrid Cloud
Combines public + private. Complex. Used when some data must stay on-prem.
On-Premises
Company owns and manages all hardware. Highest capital expense. Maximum control.

Containerization

Containers package an application together with all its dependencies, libraries, and configuration so it runs identically across development, testing, and production environments. The defining benefit: "run the same way everywhere." Containers are lighter than VMs (which run a full OS). Serverless runs code on demand but does not package dependencies for consistent deployment.

SQL Objects

View
A saved query (not stored results). Re-executes underlying SQL each time it's called. Persists permanently across sessions.
Temporary Table
Stores a result set that exists only for the current session, then is automatically dropped. "Exists only during the session."
Materialized View
Stores query results physically. Persists across sessions (unlike temp tables). Must be manually refreshed. Faster than views for repeated queries.
Index
Speeds up data retrieval on a specific column by allowing the database to find rows without a full table scan. Does not store result sets.
Slow query on a column? Always create an index on that column. This works for any frequently filtered or searched column (CustomerID, OrderDate, etc.).

Identify Artificial Intelligence (AI) Concepts

ML types, bias/fairness, model evaluation, and AI ethics

AI Fairness and Bias (Tested in Data Ethics Questions)

Fairness High Priority
When an AI model consistently favors one demographic group over another — even with no intentional discrimination — it raises a fairness concern. A model can be "accurate" in aggregate yet still be unfair to specific groups. Fairness is about equitable treatment across groups, not about overall accuracy statistics.

The root cause is often biased training data or biased feature selection, not malicious intent.
Transparency Ethics
When an organization uses customer data in ways that customers were not informed about or did not consent to — such as inferring health conditions from purchase history without disclosing this — it violates the principle of transparency. Users must be informed about how their data is collected and used.

Key AI Ethics Principles

PrincipleDefinitionExample Violation
FairnessModel treats all groups equitably; no systematic biasAI hiring tool consistently rejects one demographic group
TransparencyOrganization is open about data collection and useUsing purchase history to infer health data without disclosure
AccountabilityClear responsibility for model decisions and outcomesNo process for users to contest automated decisions
PrivacyProtecting personal data from misuse or unauthorized accessSharing health data without anonymization
Common Confusion: If a model has biased outcomes, the answer is almost always "fairness" — not "accuracy." Accuracy measures how often the model is correct overall; fairness measures whether it's equitable across groups. You can have high accuracy and poor fairness simultaneously.

Data Acquisition Methods

How to collect data — surveys, APIs, sampling strategies, integration

Qualitative vs Quantitative Data Collection

Open-Ended Surveys
Collects direct qualitative feedback from users in their own words. Use when you need opinions, sentiment, and subjective experiences. The correct choice for "qualitative insights directly from users."
Application Logs
Capture system events, errors, and technical records. Not for gathering human opinions. Qualitative feedback cannot come from logs.
API Data Pulls
Programmatic retrieval of structured data from external systems. Integration method — not for collecting user opinions.
IoT Sensor Data
Physical/environmental measurements (temperature, motion, location). Cannot capture subjective human feedback.

Sampling Methods

Sampling MethodHow It WorksKey Signal
Cluster SamplingRandomly select entire groups (clusters), then study all members within those groups"Three randomly chosen departments" — whole groups are selected, not individuals from every group
Stratified SamplingDivide population into subgroups (strata), then sample some individuals from every stratumSome from EVERY group — proportional representation across all subgroups
Systematic SamplingSelect every nth individual from an ordered listFixed interval: every 10th person, every 5th record
CensusCollects data from every single member of the entire population"Every employee," "all customers" — 100% coverage
Random SamplingEvery member has an equal chance of selectionNo bias in selection; used as a baseline
Cluster vs Stratified — The Most Common Confusion:
Cluster = select WHOLE groups randomly, study everyone in those groups. Only some groups are included.
Stratified = divide into groups, sample SOME INDIVIDUALS from EVERY group. All groups are represented.

Data Integration

Data integration is the process of combining data from multiple, disparate source systems into a unified, consistent view for analysis. Example: merging HR employee data with payroll finance data to get a complete employee picture. This is what feeds a data warehouse.

ETL (Extract, Transform, Load)
Data is extracted, transformed in a separate engine, then loaded into the destination. Transformation happens BEFORE loading.
ELT (Extract, Load, Transform)
Raw data is extracted and loaded first, then transformed inside the destination warehouse using the warehouse's computing power. Transformation happens AFTER loading.
ETL vs ELT: "Load raw data and transform within the warehouse using its computing power" = ELT. If transformation happens before loading (in a staging/processing engine) = ETL.

Data Exploration & Identifying Inconsistencies

Data quality issues, anomalies, validation, redundancy, and outliers

Data Quality Issue Types

Issue TypeDefinitionExample
Validation ErrorInput doesn't meet required format or type rules. A field rule is missing or misconfigured.Phone number field contains "555-12AB" — letters allowed in a numeric field
CompletenessData is missing (null, empty, absent)Customer records with no email address or empty date of birth
RedundancySame information stored in multiple forms, causing inconsistencyBoth "California" and "CA" stored in every record — two representations of the same value
OutlierA data point far outside the normal range of the datasetOne store reporting $150,000 when all others report $10K–$15K
DuplicationThe same record appears more than onceSame customer ID listed twice with identical information
Validation vs Normalization Confusion: When a phone field accepts letters instead of digits — that's a validation problem (missing input rule). Normalization is the solution to redundancy (a schema design technique). Normalization is a fix, not the name of the problem.

Missing Values & Imputation

Choosing the Right Imputation Method Appeared Twice (Q38 & Q65, Exam 6)
  • Mean imputation: Good for normally distributed data. Distorted by skew and outliers — the mean gets pulled toward extreme values.
  • Median imputation: The robust alternative when data is skewed. The median represents the middle value and is unaffected by extreme outliers. When the scenario mentions skew or outlier concerns, always choose the median.
  • Standardization ≠ Imputation: Standardizing (z-score scaling) rescales existing values. It does NOT fill missing values — imputation must happen first.
  • Never: Delete the entire dataset to handle missing values in one column. Drop only the missing rows if necessary, or impute.

Outliers

An outlier is a single data point that lies far outside the normal distribution of the rest of the data. Outliers always warrant investigation — they may indicate data entry errors, exceptional events, or genuinely unusual performance. A $150,000 sales figure when all others are $10K–$15K is an outlier, not a missing value, and not a normalized value.

Data Transformation & Cleansing Techniques

Reshaping, normalizing, scaling, merging, and cleaning data

Key Transformation Techniques

TechniqueWhat It DoesSignal Words
ExplodingSplits a single cell containing multiple values (comma-separated list) into one row per value. One row becomes many rows."Tags column," "split into rows," "comma-separated list," "count each category"
Normalization (Schema)Database design technique to eliminate redundancy by extracting repeated data into its own table and referencing via foreign key"Repeated data in every row," "CategoryName in every product row," "eliminate redundancy"
AggregationCombines/summarizes multiple values into one (SUM, COUNT, AVG). Opposite direction from exploding."Total," "sum," "count," "average per group"
Merging / ConcatenationCombines separate datasets with identical structure by stacking rows (appending)"Four quarterly files with same columns," "combine into annual dataset"
JoiningCombines two tables side-by-side on a shared key column (adds columns, not rows)"Match products to supplier names," "related records from two tables"
ParsingSplits a column's content into components. Does not combine files."Split full name into first/last," "extract domain from email"
BinningGroups continuous numeric values into discrete ranges/buckets"Age groups 0–18, 19–35, 36–60," "income brackets"
Exploding vs Aggregation (Common Trap): Exploding = one row becomes many (splitting). Aggregation = many rows become one (summarizing). They are opposite operations. "Split tags into rows" = Exploding. "Sum sales by region" = Aggregation.

Data Scaling Methods

Min-Max Scaling (0 to 1)
Scales all values to a 0–1 range using min and max as anchors. Extremely sensitive to outliers — a single extreme value stretches the scale and compresses all other values near 0 or 1. Worst choice when outliers are present.
Z-Score Standardization
Scales based on how many standard deviations a value is from the mean. More robust than min-max with outliers. Formula: (value – mean) / std dev.
Robust Scaling (Median/IQR)
Uses median and interquartile range instead of mean and std dev. Most resistant to outliers of all scaling methods. Ideal when data is heavily skewed.

SQL Transformation Operations

OperationPurposeKey Distinction
UNIONStacks two result sets vertically; removes duplicates (implicit DISTINCT)Same structure required; de-duplicates; performance cost
UNION ALLStacks two result sets; preserves ALL rows including duplicatesFaster than UNION — no deduplication step. Use when duplicates are acceptable.
JOINCombines two tables side-by-side on a matching key; adds columnsUse when you need data from two tables in one row
DISTINCTRemoves duplicate rows from a result setAdd to SELECT clause: SELECT DISTINCT ... FROM ...
Nested Query (Subquery)Inner query computes a value first; outer query uses that value as a filter"Calculate average first, then filter records above it"
Parameterized QueryUses placeholders instead of hardcoded values; reusable + prevents SQL injection"Reuse same query for different countries without hardcoding"

Normalization (Schema Design)

When and How to Normalize
When non-key data (like CategoryName and CategoryDescription) is repeated in every row, you have a normalization problem. The fix:
  1. Move the repeated data (category info) to its own separate table
  2. Assign it a primary key (CategoryID)
  3. Replace the repeated columns in the original table with a foreign key (CategoryID)
This eliminates redundancy and prevents update anomalies. Remember: normalization is the solution, not the problem. The problem is redundancy.

Data Governance Documentation

Parent-Child Structure
One governing "parent" document (e.g., Corporate Data Strategy) branches into multiple subordinate "child" documents (Data Storage Standards, Access Control Guidelines). Hierarchical and centralized.
Canonical Model
A single, agreed-upon data definition and structure that all source systems map to during integration. Eliminates "active customer" definition conflicts across systems. Applied during integration with mapping rules.

Communication Approaches for Data Analysis

Matching depth, format, and audience — executives, managers, operations, technical teams

The Audience-Detail Ladder

Core Rule: Match the level of detail to the audience's role and need. More detail is not always better — wrong-level detail wastes the audience's time and obscures key messages.
AudienceWhat They NeedBest FormatAvoid
C-Level Executives Strategic, high-level takeaways aligned with business goals. ROI, growth trends, key outcomes. Executive summary with key metrics and charts. Concise, no jargon. Multi-tab dashboards, line-by-line region analysis, verbatim feedback
Managers / Directors Trend summaries, department-level performance, period-over-period comparisons Monthly dashboards, summary reports with trend charts Row-level error details, raw transaction logs
Individual Contributors (Operations) Specific, actionable, row-level data. Which records failed? What error? Which field? Detailed error report with affected records, error messages, field names Bar charts of trends over time, monthly summaries, forecasting models
Technical Teams (Engineers) Backend logs, event timelines, error codes, stack traces for root-cause analysis Raw logs + event timeline of system events Infographics, plain-language summaries, customer-facing impact descriptions

Report Formats by Purpose

Ad Hoc Report
Created on demand for a single, specific question. One-time, manually generated, not on a schedule. "The VP urgently needs to know how last night's email campaign performed."
Snapshot Report Archive
Captures and freezes data at a specific point in time. Immune to retroactive edits. Used for compliance — proving what the data looked like at month-end.
Real-Time Monitoring
Continuously processes streaming data and triggers instant alerts when thresholds are crossed. "Detect intrusions as they happen." Cannot be an ad hoc or scheduled report.
Scheduled Dashboard
Refreshes at fixed intervals (daily, weekly). Shows trends over time. Not instant — introduces delay between events and visibility.
Snapshot vs Archive: Snapshot = point-in-time capture for rollback/recovery (before a major update). Archive = long-term storage of historical data no longer actively used (retention, compliance). Both preserve historical data — but snapshot is about recovery, archive is about retention.

Operational Report Content

SLA Breach Reports (Operational Team Leads) Exam 6
When a support team lead needs to identify which tickets breached SLA and which queue caused delays: the correct answer is a detailed internal report with SLA KPIs, queue-level breakdowns, and exception lists. Executive summaries strip away the operational detail that the team lead needs to act on. Research papers are for academic audiences.

Statistical Methods & Functions

Analytics types, measures of center and spread, regression, optimization, functions

Analytics Types — The Most Tested Hierarchy

The Four-Level Framework (must memorize):
Descriptive → Diagnostic → Predictive → Prescriptive
1. Descriptive Analytics
What happened? Summarizes historical data. Averages, totals, trends on existing data. "Show average monthly revenue across the year." Uses: pivot tables, dashboards, summary statistics.
2. Diagnostic Analytics
Why did it happen? Root cause analysis. Examining past data to understand the cause of an observed outcome. Uses: drill-down reports, correlation analysis.
3. Predictive Analytics / Modeling
What will happen? Forecasts future outcomes using historical data. Regression, classification models. "Estimate next quarter's import volumes."
4. Prescriptive Analytics
What should we do? Recommends the optimal action under constraints. Optimization algorithms. "Best price and budget combo to maximize profit." "Most efficient delivery routes."
Prescriptive Signal Words: "best combination," "maximize," "minimize," "optimize," "most efficient," "recommend action." If the question asks what decision to make, the answer is prescriptive. If it just asks what will happen, it's predictive. If it summarizes what already happened, it's descriptive.

Measures of Center (Central Tendency)

MeasureWhat It DescribesWhen to Use
MeanArithmetic average of all valuesNormally distributed data; no significant outliers
MedianMiddle value of an ordered datasetSkewed data or when outliers are present; more robust than mean
ModeMost frequently occurring valueCategorical data; nominal data; identifying most common value
Critical Distinction: Standard deviation and variance measure spread (how far values are from the mean) — NOT central tendency. "Typical value" or "central value" always means mean, median, or mode. Never standard deviation.

Measures of Spread (Dispersion)

MeasureDefinitionStrength
RangeMax minus min. Crude measure.Simple; highly distorted by a single extreme value
IQR (Interquartile Range)Q3 minus Q1 — spread of the middle 50% of dataRobust; not affected by outliers. Best for comparing consistency.
VarianceAverage squared deviation from the meanMeasures spread; hard to interpret (squared units)
Standard DeviationSquare root of variance; same units as dataMost commonly used spread measure; sensitive to outliers

Statistical Methods for Predictions

Logistic Regression
Predicts the probability of a binary outcome (yes/no, filed/not filed). Use for: "likelihood of a claim being filed," "will a customer churn." Binary probability forecasting.
Linear Regression
Models the relationship between variables to predict a continuous outcome. "Predicted sales based on advertising spend."
Optimization Algorithm
Finds the best solution from possible options given constraints and objectives. "Most efficient delivery route minimizing fuel and time" (Travelling Salesman Problem type). This is prescriptive.
Hypothesis Testing
Evaluates whether an observed effect is statistically significant. Tests a claim about existing data — does NOT generate probability forecasts.

Troubleshooting Tools & Methods

Diagnosing report errors, SQL errors, connector issues, and stale data

Common Troubleshooting Scenarios

SymptomMost Likely CauseSolution
Report shows last month's data even though database was updated Report was not refreshed after the database update (cached snapshot) Manually refresh the report, or check refresh settings
Report shows no data at all with an error Invalid connection string (report cannot connect to data source) Verify the connection string; a connection error causes failure, not stale data
Dashboard takes several minutes to refresh Live queries against multiple large datasets on every refresh Convert live connections to scheduled extracts (pre-cached snapshots)
Third-party connector fails with unknown error code Connector-specific issue not covered in internal documentation Search the vendor's online knowledge base or community using the error code before escalating
SQL query returns "syntax error near FROM" SELECT and FROM clauses are in the wrong order Correct order: SELECT → FROM → WHERE → GROUP BY → HAVING → ORDER BY
Stale Data Rule: An invalid connection string causes an error (no data at all). Stale/old data with no error message = the report was not refreshed. These two symptoms have different causes.

Appropriate Visual Elements

Chart types, design principles, pivot tables, and when to use each

Visualization Selector — Know Which Chart to Use

Chart TypeBest ForNOT For
Bar ChartComparing values across categories; showing differences between groupsPerforming the grouping/aggregation itself; geographic data
Line ChartTrends over time; continuous data over a sequenceComparing categories at a single point in time; geographic distribution
Pie ChartParts of a whole at a single point in time (2–5 categories max)50 states comparison; time trends; multi-dimensional breakdowns
Choropleth MapShowing how a metric varies across geographic regions (states, countries)Non-geographic data; trend over time
Heat MapIntensity values in a matrix grid formatGeographic boundaries — heat maps use grids, not real map shapes
Gantt ChartProject schedules and task timelinesPerformance metrics; geographic data; comparing values by category
Scatter PlotRelationship between two continuous variables; correlationCategorical comparisons; time trends

Pivot Table Rule: When a question asks how to group data by categories and calculate aggregates, the answer is a pivot table — not a bar chart. A bar chart displays pre-aggregated results. A pivot table performs the grouping and aggregation. "Group employees by department and calculate average hours" = pivot table.
Geographic Data = Choropleth Map: "Visualize number of users by state across the U.S." → Choropleth map. Pie chart loses geographic context. Heat map uses a grid, not geographic boundaries.

Design Principles for Cluttered Reports

Improving Report Readability
When a report is visually cluttered, the primary fix is to use consistent color schemes — this reduces visual noise and helps readers identify trends. Incorrect approaches include:
  • Removing all gridlines — can hurt readability by eliminating reference points
  • Increasing data granularity — adds more clutter, not less
  • Changing the data source — changes what is shown, not how readable it is

Data Flow Diagrams (DFD)

Data Flow Diagram (DFD)
Maps how data moves between external entities, processes, and data stores. Used to document data movement across systems (collection → processing → storage → output).
Entity Relationship Diagram (ERD)
Shows the structure of a database — entities, attributes, and relationships between tables. Documents data structure, not data movement.
Process Flowchart
Documents the steps of a business process or workflow. Focuses on process logic — not data movement between systems and storage locations.

Report Delivery Methods

Delivery MethodDescriptionUse Case
Ad Hoc ReportCreated on demand, one-time, for a specific question. Not scheduled."VP needs urgent one-time analysis of last 24 hours"
Snapshot ArchiveFrozen point-in-time data capture. Immune to retroactive changes.Compliance: month-end figures that must not change when source data is updated

Scheduled DashboardRefreshes at fixed intervals (daily, weekly). Not real-time.Regular management reporting, performance tracking
Scheduled ExtractPre-processes and caches data on a schedule so users get fast snapshots instead of live queriesFixing slow live-data dashboards — converts to pre-built snapshots
Recurring ReportAutomatically generated on a fixed schedule (e.g., weekly)Regular automated distribution to a standard audience
Snapshot vs Archive (Remixed):
Snapshot (report) = frozen data for compliance purposes (cannot change retroactively).
Snapshot (database) = point-in-time capture for rollback before a major system change.
Archive = long-term historical storage of data no longer actively used. Purpose is retention, not recovery or compliance freezing.

Data Privacy & Protection Practices

Regulations by jurisdiction, anonymization, encryption, access control, and data minimization

Global Privacy Regulations — Match the Regulation to the Country

RegulationJurisdictionScope
GDPREuropean UnionPersonal data of EU residents; applies regardless of where the company is located
LGPDBrazilPersonal data of Brazilian residents; modeled after GDPR; applies regardless of where servers are
CCPACalifornia, USAPersonal data of California residents
HIPAAUnited States (Federal)Healthcare data only — medical records, PHI (Protected Health Information)
PCI DSSGlobal (payment industry)Payment card data; requires encryption at rest using strong encryption; never store CVV
Jurisdiction Rule: Always match the regulation to where the data SUBJECT lives, not where the server is. Brazilian customer data on US servers = LGPD. EU customer data = GDPR. California customer data = CCPA. US healthcare data = HIPAA.

Privacy Techniques

TechniqueWhat It DoesReversible?
AnonymizationIrreversibly removes personal identifiers so individuals cannot be re-identifiedNo — permanent. "Cannot be re-identified" = anonymization.
PseudonymizationReplaces identifiers with pseudonyms; original identity can be recovered with a keyYes — reversible with the key
Data MaskingObscures values (preserving format) for display/testing purposes. Often retains structure.Partially — depends on method; not the same as anonymization
EncryptionScrambles data using a key; original can be recovered with the keyYes — reversible with key. PCI DSS requires this for stored card data.
Data MinimizationCollect and share only the data necessary for the intended purpose; remove what's not neededN/A — a principle of collection/sharing, not a transformation
"Cannot be re-identified" = Anonymization (irreversible).
Encryption scrambles but is reversible with a key — it does not make individuals unidentifiable.
Data masking preserves format and is not truly irreversible in all implementations.

PCI DSS Requirements (Payment Card Data)

What PCI DSS Requires Common Exam Question
  • Encrypt cardholder data at rest using strong encryption methods (AES-256 or equivalent)
  • Never store the CVV after authorization under any circumstances
  • Never store data in plaintext — even partial plaintext (first 6 + last 4 visible, rest in plaintext) violates PCI DSS
  • MD5 is NOT acceptable for protecting card data — it's a broken, insecure algorithm
  • Mask the PAN (card number) when displayed — show only first 6 and last 4 digits maximum

Access Control

Role-Based Access Control (RBAC) Separation of Duties
RBAC restricts what each user can see and do based on their job role. Assigning support agents a role that grants access to order status but not payment details is the direct mechanism for enforcing separation of duties at the data level.

What RBAC is NOT:
  • Disk encryption — protects data at rest from external access; does not restrict authenticated users
  • HTTPS — protects data in transit; does not control what a logged-in user can see
  • Masking all customer names — addresses the wrong field; doesn't restrict by role
