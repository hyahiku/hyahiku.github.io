01

Overview & Motivation

The Business Problem

Businesses, governments, and other organizations routinely handle the Personally Identifiable Information (PII) of customers, employees, and vendors. When that PII leaks or is transmitted where it shouldn't be, the legal, financial, and reputational consequences are severe — and the threat landscape is growing as attackers adopt more capable tooling of their own.

The Proposed Solution

An in-house model that scans documents, emails, and other correspondence and flags whether they contain PII. Anything flagged can be masked or blocked before it leaves the organization. The goal of this project is to prove the concept with a fully transparent network — every weight and gradient hand-derived — rather than a black-box framework.

Why build it from scratch?

Frameworks hide the mechanics. Implementing forward propagation, the loss, and backpropagation by hand forces every design choice — initialization, activation, regularization, the gradient of the loss — to be explicit and defensible. The result is a model whose behavior can be explained line by line, which matters for a compliance-adjacent use case.

Why this matters — recent breaches

The cost of getting PII handling wrong is not hypothetical. Two 2024 incidents illustrate the scale:

192.7M Change Healthcare · 2024

People whose PII was exposed in a single healthcare-sector breach.

2.9B National Public Data · 2024

Records leaked, including Social Security numbers and other personal details.

Dataset & Preparation

The source data

The project uses the English subset of the public ai4privacy / pii-masking-300k dataset, loaded directly from the Hugging Face Hub. Each row pairs a source_text snippet with a privacy_mask field listing any PII entities found within it, alongside token-level annotations.

Defining the label

The task is reframed as a simple binary problem: a row is labeled 1 (has PII) if its privacy_mask contains at least one entity, and 0 (no PII) otherwise. This collapses rich span annotations into the single signal the network needs to learn.

The class-imbalance problem

The raw data is heavily skewed — nearly 9 in 10 rows contain PII. A model trained on it as-is could reach ~88% accuracy by always predicting "PII" while learning nothing useful. So the data was rebalanced to a 1:1 ratio before training.

The balancing strategy — five cycles

Rather than discard most of the PII examples, the 26,426 PII rows were shuffled and split into five equal partitions of 3,482 — matching the count of non-PII rows exactly. Each "cycle" pairs one PII partition with the full set of non-PII rows, yielding a perfectly balanced training set of ~6,964 examples. Across all five cycles the model can eventually see every PII example without ever training on imbalanced data.

Vectorization & Embeddings

A neural network consumes numbers, not text. Turning each document into a fixed-length numeric vector was the single most consequential design decision in the project — and the first approach had to be abandoned.

First attempt — TF-IDF (and why it failed)

The initial approach used TF-IDF: each document became a sparse vector over a vocabulary of unigrams and bigrams, weighting rare, distinctive terms heavily. In principle this suits PII, where the signal often lives in unusual tokens like emails, dates, and ID numbers. In practice, the vectors were extremely sparse and high-dimensional, and the from-scratch network would not learn from them.

The symptom

Training loss sat frozen at 0.6931 — exactly ln(2), the loss of a coin-flip — and accuracy hovered at ~49–50% across every epoch. The logits stayed near zero everywhere: with thousands of mostly-zero features feeding a hand-coded network, gradients had effectively nothing to propagate. The math was correct; the representation was the problem.

The fix — averaged GloVe word embeddings

The representation was switched to pre-trained GloVe 6B 300d embeddings from Stanford — 400,000 word vectors trained on six billion words. Each document is converted to a single dense 300-dimensional vector by averaging the GloVe vectors of all its words. Documents with no recognized words fall back to a zero vector.

# Average the GloVe vectors of every word in a document → one 300-d vector
def text_to_embedding(text, embeddings_index, dim=300):
    words = text.lower().split()
    valid = [embeddings_index[w] for w in words if w in embeddings_index]
    if len(valid) == 0:
        return np.zeros(dim, dtype=np.float32)   # no known words
    return np.mean(valid, axis=0)

Result: X_emb shape (45,000, 300) — dense, low-dimensional, semantically meaningful.

Why this works where TF-IDF didn't

Averaged embeddings collapse extreme sparsity into a compact, dense vector that already encodes semantic meaning learned from billions of words. Instead of asking the network to discover structure in a 20,000-dimensional sea of zeros, it receives 300 informative dimensions — far easier for a small NumPy network to learn from, and dramatically faster to train.

6B words

GloVe was trained on a six-billion-word corpus, giving each vector a strong grasp of general English.

300 dims

Each word maps to a rich 300-dimensional representation — detailed without being unwieldy.

~822 MB

A manageable download that is simple and fast to work with on general English text.

05

Model Architecture

ScratchNet is a feed-forward neural network for binary classification, implemented entirely in NumPy. It takes a 300-dimensional embedding and outputs the probability that the text contains PII.

Fig. 2 — ScratchNet: a 300→256→128→1 fully-connected network. Hidden layers use ReLU with dropout; the output neuron's logit is passed through a sigmoid to produce P(has PII).

Design choices, and the reason for each

He initialization

Hidden weights are drawn from a normal distribution scaled by √(2 / fan-in). This is the standard pairing for ReLU networks — it keeps the variance of activations stable across layers so signals neither vanish nor explode early in training.

ReLU activation

max(0, z) on hidden layers. It is cheap to compute, has a clean derivative (1 where active, 0 otherwise) that is trivial to hand-code in backprop, and avoids the saturation that would stall gradient flow.

Inverted dropout

During training, a random fraction of hidden activations are zeroed and the survivors are scaled up. This regularizes the network against over-fitting and is switched off entirely at inference time.

Sigmoid output

A single output neuron produces a logit, mapped by the sigmoid to a probability in (0, 1). A threshold of 0.5 turns that probability into the final has-PII / no-PII decision.

The loss — numerically stable binary cross-entropy

The network learns by minimizing binary cross-entropy. Rather than computing the sigmoid and then its log — which overflows for large logits — the loss is computed directly from the raw logits in a numerically stable form that never evaluates a near-zero logarithm.

Stable BCE loss
L = mean[ max(z, 0) − z·y + log(1 + e−|z|) ]

Mathematically equivalent to standard BCE but stable even for |z| > 50. Crucially, its gradient with respect to the logit is simply sigmoid(z) − y — the clean starting point for backpropagation.

The four methods that make it learn

1

forward()

Passes embeddings through each layer, applying ReLU and dropout, and returns raw logits. It caches every activation and dropout mask for use in the backward pass.

2

backward()

Computes gradients by backpropagation. It starts from sigmoid(z) − y and works backward layer by layer, deriving dW and db for each — applying the ReLU derivative and dropout mask along the way.

3

update_parameters()

Applies a stochastic gradient descent step with L2 weight decay: W ← W − lr·(dW + λ·W). The decay term penalizes large weights, discouraging over-fitting.

4

predict_proba()

Switches dropout off, runs a forward pass, and returns the sigmoid probability that the text contains PII.

06

Training

One training cycle ties the four methods together into the loop that every neural network runs: predict, measure error, assign blame, adjust.

# Core learning loop — one balanced cycle
for epoch in range(epochs):           # 60 epochs per cycle
    logits = model.forward(X_train)           # 1. predict
    loss   = stable_bce_loss(logits, y_train) # 2. measure error
    model.backward(logits, y_train)         # 3. assign blame (gradients)
    model.update_parameters(lr=lr)          # 4. adjust weights (SGD + L2)

SGD + L2

Vanilla stochastic gradient descent with L2 weight decay (λ = 0.001) as the optimizer — no momentum, no Adam, kept deliberately transparent.

60 epochs

A single demonstration cycle runs 60 full passes over the balanced data, after which the learned weights are saved to disk.

Save & reuse

Trained weights are persisted so that inference testing runs independently — load weights, predict, evaluate.

From a stuck loss to a learning curve

Under TF-IDF the loss never moved from 0.6931. After switching to averaged GloVe embeddings — the same network, the same hand-coded backprop — the loss began to fall and accuracy climbed. The lesson the project surfaced repeatedly: when a correct network won't learn, suspect the input representation before the math.

Reading the result honestly

83% overall accuracy from a hand-coded network with no autograd is a genuine proof of concept. But the per-class view matters more than the headline: the model is strong on "No PII" (F1 0.88) and weaker on the minority "Has PII" class (F1 0.71). With only 8 positive examples in this test set, every error moves the numbers a lot — and for a compliance tool, missing real PII (false negatives) is the costlier mistake to watch.

08

Conclusion & Next Steps

// 01

The math works by hand

A network with manually derived forward and backward passes — and no deep-learning framework — learned a real text-classification task to 83% accuracy. The mechanics of deep learning are tractable in plain NumPy.

// 02

Representation beats architecture

The same network failed on TF-IDF and succeeded on averaged GloVe embeddings. The biggest gain came from feeding the model a better representation, not from changing its layers.

// 03

Balance the data first

An 88/12 split would have produced a deceptively "accurate" but useless model. Rebalancing to 1:1 across five cycles was what made the accuracy meaningful.

// 04

Real business value

An in-house screen that flags PII in documents and email lets an organization mask or block sensitive data before it leaves — directly reducing legal, financial, and compliance exposure.

Limitations & future directions

Larger test set

Thirty test examples is too few to trust precise metrics. Evaluating across all five cycles' held-out data would give far more stable estimates.

Multi-class detection

Extend beyond binary to distinguish PII types — payment-card (PCI), health (HIPAA), and other protected categories — for finer-grained policy enforcement.

Recall-first tuning

For compliance, false negatives are the dangerous error. Lowering the decision threshold or weighting the loss toward the positive class would prioritize catching PII over avoiding false alarms.
