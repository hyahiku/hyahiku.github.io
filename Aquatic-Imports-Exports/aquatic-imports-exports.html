01 · Overview

What Was Built & Why

Project Context

This project builds on a prior data cleaning exercise using the same Global Aquaculture Imports and Exports dataset. Where the previous project focused on cleaning and basic descriptive statistics — averages of revenue and import/export rates — this project goes further: forecasting future revenue using historical trade data and building an interactive query interface for any country in the dataset.

The dataset was provided as 12 separate CSV files, each covering a specific region and trade metric type. The first engineering challenge was consolidating these files into a unified, analysis-ready structure.

Core Output

The final product is an interactive command-line tool. A user enters any country name and receives a complete trade profile — total imports and exports, annual averages, cumulative revenue, revenue rank relative to global average, peak/low revenue years, and a 5-year growth forecast — all generated on the fly from the underlying data.

Each piece of the output is produced by a dedicated function. This modular design means the system is composable — any individual statistic can be extracted independently, and the full profile is assembled by calling all functions together.

02 · Dataset

12 CSVs → 2 Unified DataFrames

The source data consists of 12 CSV files — one Quantity file and one Value file for each of six regions: Africa, Americas, Asia, Europe, Global, and Oceania. Each file shares a consistent column structure: Land Area (country), Trade Flow (import/export), Commodity (fish/crustaceans), and year columns from 2000 to 2015.

Quantity Files (×6)

Record how much fish was traded (volume units) per country per year, split by import and export. Combined into a single Q_data DataFrame covering all regions.

Value Files (×6)

Record the monetary value of traded fish (USD) per country per year. Combined into a single V_data DataFrame. This is the primary input for revenue analysis and forecasting.

Data Processing Pipeline — process_data()

A single reusable function handles the consolidation and cleaning of all six regional files into one unified DataFrame. The same function is called twice — once for Quantity files, once for Value files.

Step 1 · Merge

pd.concat() combines all six regional DataFrames along rows into a single merged DataFrame, resetting the index. This produces one comprehensive table covering all countries across all regions.

Step 2 · Drop Columns

Columns beginning with capital 'S' are removed — these are source/flag columns that are not needed for analysis and would add noise to the dataset.

Step 3 · Rename

The 'Land Area' column is renamed to 'Country' for readability and consistent referencing throughout the analysis functions.

Step 4 · Filter Commodity

Rows where Commodity = 'Crustaceans' are removed. The analysis focuses exclusively on fish trade — keeping crustaceans would mix two fundamentally different commodity categories and distort aggregated totals.

Step 5 · Remove Zeros

Rows where any year column (those starting with '2') contains a zero value are dropped. Zero entries represent missing or unreported data rather than genuine zero-trade years, and would distort averages and totals.

Step 6 · Type Cast

All year columns are explicitly converted to integer type. This ensures consistent numeric behavior for summation, averaging, and regression calculations downstream.

def process_data(csv_files):
    merged_df = pd.concat(csv_files, ignore_index=True)
    merged_df = merged_df.loc[:, ~merged_df.columns.str.startswith('S')]
    merged_df = merged_df.rename(columns={'Land Area': 'Country'})
    merged_df = merged_df[merged_df['Commodity'] != 'Crustaceans']
    relevant_cols = [col for col in merged_df.columns if col.startswith('2')]
    merged_df = merged_df[(merged_df[relevant_cols] != 0).all(axis=1)]
    for col in relevant_cols:
        merged_df[col] = merged_df[col].astype(int)
    return merged_df

03 · Analytical Architecture

Modular Function Design

Rather than writing a single monolithic script, the analysis was structured as a library of focused, reusable functions. Each function accepts a country name and the relevant DataFrame as inputs, performs one specific calculation, and returns a formatted string. This makes the system composable — the full country profile is built by calling each function in sequence.

This design pattern mirrors real-world data product architecture: isolating logic into single-responsibility functions makes the code testable, readable, and extensible. Adding a new statistic means adding one new function, not rewriting existing code.

04 · Functions

Eight Core Functions

import_sum()
country, Q_data → str
Filters Q_data to rows where Country matches and Trade Flow = 'Import'. Sums all values across the 16 year columns. Returns a comma-formatted integer string of total fish imported 2000–2015.
export_sum()
country, Q_data → str
Same logic as import_sum() but filters for Trade Flow = 'Export'. Returns total fish exported across all years as a formatted string.
import_avg()
country, Q_data → str
Calculates total import quantity then divides by 15 (the number of years in the dataset) to produce an average annual import figure. Returns as a comma-formatted string.
export_avg()
country, Q_data → str
Mirrors import_avg() for export trade flow. Divides total export quantity by 15 to produce the average annual export volume.
value_sum()
country, V_data → str
Sums all monetary values in V_data for a given country across all year columns — both imports and exports combined. Returns the total USD trade value over the full 15-year period.
money_rank()
country, V_data, V_average → str
Compares the country's total value against a pre-computed global average (V_average). Returns one of three strings: "more than", "less than", or "about" — providing a relative position in the global trade value distribution.
lowest_year()
country, V_data → str
Iterates through all year columns to find the one with the minimum trade value for the given country. Returns the year column name (e.g. '2003') — the country's lowest revenue year.
highest_year()
country, V_data → str
Mirrors lowest_year() to identify the peak revenue year for a given country across the 2000–2015 window.
forecast_growth()
country, V_data → int
Fits a linear regression to the country's 16-year revenue series. Projects revenue for the next 5 years (indices 16–20) and returns their sum as a formatted integer — the expected total growth over the forecast window.

05 · Forecasting

Linear Regression Revenue Forecast

The core machine learning technique in this project is linear regression applied to each country's revenue time series. The model is fit on 16 years of historical data (2000–2015) and projected forward 5 years to estimate trade value through 2020.

1
Extract Historical Series

For a given country, total trade value is summed across all rows per year column — producing a 16-point time series where each point is the country's total trade value in that year.

2
Fit Linear Regression

np.polyfit(x, y, 1) fits a first-degree polynomial (straight line) to the historical data, returning slope m and intercept b for the equation y = mx + b. The x-axis uses integer indices (0–15) rather than calendar years to simplify the arithmetic.

3
Project Forward 5 Years

Future indices 16 through 20 are fed into the fitted equation. The resulting five predicted values represent annual trade value estimates for 2016–2020. Their sum is the "growth forecast" displayed in the output.

4
Visualize

visualize_growth() plots the historical values as a connected line, extends it through the forecast period, and marks the 2015 boundary with a vertical dashed line — clearly distinguishing observed data from projected data.

def forecast_growth(country, V_data):
    year_cols = [col for col in V_data.columns if col.startswith('2')]
    values = [V_data[V_data['Country'] == country][col].sum() for col in year_cols]

    x = np.arange(len(values))          # indices 0–15
    y = np.array(values)
    m, b = np.polyfit(x, y, 1)           # fit y = mx + b

    future_x = np.arange(len(values), len(values) + 5)  # indices 16–20
    forecast = np.sum(m * future_x + b)  # sum of 5 projected years
    return round(forecast)

Why linear regression? With 16 years of data and a clear upward trend evident across most countries, a simple linear model is the appropriate first approach. It is interpretable, fast to compute per-country, and sufficient to capture the dominant growth signal. More complex models — time series (ARIMA/SARIMA) or polynomial regression — could better capture non-linear trends and seasonality, but are outside the scope of this exploratory project.

06 · Conclusion

What This Project Demonstrates

Each country in the dataset shows linear growth in trade revenue over the 2000–2015 window — though the variance in that growth differs significantly by country. The interactive interface makes the analysis accessible: instead of reading a static table, a user gets a narrative summary tailored to their country of interest, with a visual forecast appended.

Data Engineering

Consolidating 12 heterogeneous CSV files into two clean, unified DataFrames through a single reusable pipeline function. This pattern — abstract the cleaning logic, apply it to multiple inputs — is fundamental to scalable data workflows.

🔧

Modular Function Architecture

Eight dedicated functions each handle one statistic. The full output is assembled by composing them — making the codebase easy to extend, debug, and reuse. Adding a new metric means adding one function, not modifying existing logic.

📈

Applied Linear Regression

Fitting a regression model per country on time-indexed revenue data to generate forward-looking estimates. Demonstrates how even a simple model adds practical value — converting historical data into an actionable forecast.

💻

Interactive CLI Design

The while-loop input system handles valid entries, invalid entries, random country selection, and graceful exit. Building user-facing interfaces on top of analytical pipelines is a transferable pattern for any data product.

Future Directions

Linear regression assumes a constant rate of change. Future analyses could apply time series models (ARIMA, SARIMA) to capture seasonality and autocorrelation, or polynomial regression to model non-linear growth trajectories. Country-specific analysis at finer granularity could also yield more targeted forecasts for high-trade nations.
